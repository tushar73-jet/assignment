# 3-Agent Incident Response System

This repository contains a working 3-agent incident response system designed to process API server logs, research solutions, and formulate a step-by-step resolution plan.

## Architecture

The system implements the following 3 agents:

1. **Log Analysis Agent (Agent 1):**
   - **Responsibility:** Ingests `nginx-access.log`, `nginx-error.log`, and `app-error.log`. It identifies the likely root cause (Database connection pool exhaustion).
   - **Implementation:** Uses `gemini-2.5-flash` with a structured `pydantic` schema (`LogAnalysisHandoff`) to guarantee a precise JSON output including the root cause, evidence, confidence, and recommended search queries for Agent 2.

2. **Solution Research Agent (Agent 2):**
   - **Responsibility:** Performs real-world solution discovery based on the root cause without relying on an LLM for the initial discovery.
   - **Implementation:** Uses `duckduckgo_search` (`ddgs`) to scrape search results (StackOverflow, documentation, etc.) using queries generated by Agent 1. It then uses the LLM *only* to compare the scraped solutions, extract pros/cons, and flag risks into a `ResearchHandoff` format. (Note: A fallback simulated context is provided if network restrictions block the scraping).

3. **Resolution Planner Agent (Agent 3):**
   - **Responsibility:** Reviews the diagnosis and researched solutions, selecting the safest path forward.
   - **Implementation:** Prompts `gemini-2.5-flash` with Agent 1 and Agent 2's JSON outputs to formulate a step-by-step `ResolutionPlan`, including pre-checks, remediation steps, post-fix validation, and rollback procedures.

## Design Decisions and Production Reasonableness

- **Strict Handoffs:** Pydantic schemas paired with Gemini's `response_schema` feature are used to enforce structured JSON outputs. This prevents downstream agent parsing failures.
- **Agent Boundaries:** Agent 2 strictly uses web scraping for discovery. This mitigates hallucination risks because the LLM is restricted to formatting the specific search results retrieved from DuckDuckGo.
- **Safety First:** Agent 3 is explicitly prompted to select the *safest* solution and mandate pre/post-checks, ensuring that remediation advice aligns with standard SRE best practices (e.g., verifying connection pool metrics before increasing limits).
- **Graceful Degradation:** If Agent 2's search fails (e.g., due to network limits or CAPTCHAs), it falls back to a simulated set of results to ensure the pipeline continues executing.
